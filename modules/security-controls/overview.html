# Module: Security Controls & Architecture

📚 Theory & Class Notes

Security Controls are the defensive countermeasures employed to detect, deter, or mitigate cyber risks. A robust architecture uses "Defense-in-Depth"—layering multiple security controls so if one fails, others remain.

Key Deployments: Web Application Firewalls (WAF) to block SQLi/XSS, Intrusion Detection Systems (IDS), Endpoint Detection & Response (EDR), and Security Information & Event Management (SIEM).
Tools: ModSecurity (WAF), Snort, Splunk, CrowdStrike.

1. Core Concept 🧠

Security Controls are the safeguards or countermeasures implemented to avoid, detect, counteract, or minimize security risks to physical property, information, computer systems, or other assets.

Real-world analogy: Pentesting is finding the holes; Security Controls are the bricks, mortar, alarms, and guards put in place to ensure those holes are closed and monitored.

2. Control Categories 🎯

🛡️ Preventive Controls (Stop the attack)
↓
🕵️ Detective Controls (Identify the attack)
↓
🛠️ Corrective Controls (Fix the attack damage)

Key Enterprise Architectures

Control Type Implementation Effect on Attacker
Identity Control MFA & RBAC (Role-Based Access Control) Password cracking is useless without the 2FA token. Limits lateral movement.
Network Control VLAN Segmentation & Firewalls Prevents pivoting. Attacker on Guest WiFi cannot touch the Domain Controller.
Endpoint Control MDM & EDR (Crowdstrike/Defender) Blocks malicious payloads; isolates infected machines instantly.

VAPT is Reactive: Security Controls are what stand between attackers and your data when software flaws inevitably slip through the cracks. In this lab, we test against a globally recognized Web Application Firewall: ModSecurity.

Setup

# We have deployed an Apache webserver strictly protected by ModSecurity Core Rule Set (CRS).
# Make sure the lab container is running:
cd ../../labs/docker/
docker compose up -d security-controls-app

Attack Steps

  1. Normal Request: Visit the application normally in your browser at http://localhost:8002/. Notice that it loads perfectly fine (it reverse-proxies example.com safely).
  2. Malicious Request (SQL Inection): Now, try appending a classic SQL Injection payload to the URL bar:
    http://localhost:8002/?id=1' OR 1=1--
  3. Observe the Control: Press Enter. The WAF will instantly intercept the maliciously-crafted HTTP GET request. Instead of crashing a database, you receive a stark 403 Forbidden response generated locally by the firewall engine.
  4. WAF Evasion (Optional Challenge): Real-world attackers spend days trying to obfuscate payloads (using encoding, HTML entities, or weird whitespace) to bypass regular expression filters without losing functionality. Try experimenting!

Expected Output

The ModSecurity engine catches the signature of the SQLi attack and terminates the connection before the backend Web Server even knows it happened.

Forbidden
You don't have permission to access this resource.
