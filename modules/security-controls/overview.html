# Module: Security Controls & Architecture

1. Core Concept 🧠

Security Controls are the safeguards or countermeasures implemented to avoid, detect, counteract, or minimize security risks to physical property, information, computer systems, or other assets.

Real-world analogy: Pentesting is finding the holes; Security Controls are the bricks, mortar, alarms, and guards put in place to ensure those holes are closed and monitored.

2. Control Categories 🎯

🛡️ Preventive Controls (Stop the attack)
↓
🕵️ Detective Controls (Identify the attack)
↓
🛠️ Corrective Controls (Fix the attack damage)

Key Enterprise Architectures

Control Type Implementation Effect on Attacker
Identity Control MFA & RBAC (Role-Based Access Control) Password cracking is useless without the 2FA token. Limits lateral movement.
Network Control VLAN Segmentation & Firewalls Prevents pivoting. Attacker on Guest WiFi cannot touch the Domain Controller.
Endpoint Control MDM & EDR (Crowdstrike/Defender) Blocks malicious payloads; isolates infected machines instantly.

3. Secure SDLC 🧪

📋 Concept Check: Shifting Left

Topic: Defense Difficulty: Theory Time: 15 mins
🚀 Start Lab Environment
⚠️
VAPT is Reactive: If you are finding SQL Injection during a penetration test right before launch, you have failed the Secure SDLC process. Security must start at the code design phase.

Shift-Left Pipeline Concept

Key Phases

  1. Design: Threat Modeling (STRIDE) before any code is written.
  2. Develop: IDE Linters detecting XSS/SQLi in real-time.
  3. Build: SAST (Static Application Security Testing) like SonarQube analyzing source code.
  4. Test: DAST (Dynamic Application Security Testing) running automated attacks against staging.
  5. Deploy: WAF (Web Application Firewall) protecting the live application.

Expected Outcome

By implementing these controls, organizations drastically reduce the number of exploitable flaws that make it to production, making the eventual manual Pentest much harder (and less expensive) to conduct.
