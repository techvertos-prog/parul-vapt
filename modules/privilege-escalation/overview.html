Module: Privilege Escalation Deep Dive

📚 Theory & Class Notes

Privilege Escalation is the act of exploiting a bug, design flaw, or configuration oversight in an operating system or software application to gain elevated access to resources that are normally protected from an application or user.

Vulnerabilities: Sudo misconfigurations (sudo -l), SUID bit binaries, Cron jobs running as root, insecure file permissions, and unpatched kernel vulnerabilities.
Tools: LinPEAS, WinPEAS, GTFOBins, exploit-db.

1. Core Architectural Concept 🧠

Privilege Escalation (PrivEsc) is the critical bridge between gaining an initial foothold and taking total control of a server. Simply obtaining a web-shell as the www-data user is rarely the end goal; attackers almost always want to become root (Linux) or SYSTEM / Administrator (Windows) to pivot further into a network or exfiltrate core database credentials.

At an OS level, processes inherit the permissions of the user who executes them. However, Linux implements Special Permissions like SUID (Set owner User ID up on execution). When a binary has the SUID bit set, it runs with the privileges of the file's owner rather than the executor. If the owner is root, the binary runs as root. If that binary has features that allow executing external commands (like `find`, `vim`, `bash`), attackers can escape the binary's context to spawn a root shell.

💡
Pro Tip The holy grail for finding Linux PrivEsc vectors is GTFOBins (A curated list of Unix binaries that can be used to bypass local security restrictions).

2. Multi-Vector Attack Flows 🎯

Privilege escalation can be horizontal (gaining access to another standard user) or vertical (gaining administrative access). Here are three different real-world attack vectors to escalate vertically on Linux.

Vector A: SUID Binary Abuse (The Lab Focus)

👨💻 Low-Priv Shell
↓
🔍 Hunt for SUIDs (`find / -perm -u=s`)
↓
💉 Exploit `find` or `vim` binary
↓
👑 root shell via spawned subprocess

Vector B: Misconfigured Cron Jobs

👨💻 Low-Priv Shell
↓
🔍 Read /etc/crontab or run `pspy`
↓
💉 Find script running as root that is writable
↓
👑 Overwrite script with reverse shell

Vector C: Sudo Rights Misconfiguration

👨💻 Low-Priv Shell
↓
🔍 Check rights with `sudo -l`
↓
💉 Find `sudo (ALL) NOPASSWD: /usr/bin/python`
↓
👑 `sudo python -c 'import os; os.system("/bin/bash")'`

3. Hands-on Lab 🧪

🧪 Lab: Linux SUID Abuse & Sudo Misconfigs

Topic: Linux PrivEsc Difficulty: Medium Time: 45 mins
[System] Awaiting container initialization...
📋
Prerequisites: Basic Linux command line knowledge. A local terminal.

Phase 1: Establishing the Foothold

Before escalating privileges, you must connect to the compromised box. In this lab, we simulate that you have just cracked an SSH password.

# Connect to the target machine using the compromised credentials
ssh pwnme@localhost -p 2222
# password: pwnme_password
🔍
Deep Dive Analysis: Always establish your baseline. Once connected, run id and whoami. Notice you are standard user 1000. Next, check the kernel version using uname -a. If it is old, you might be able to use a kernel exploit like DirtyCOW.

Phase 2: The SUID Attack Vector

  1. Enumerate: Search the entire file system for binaries with the SUID bit set, ignoring permission denied errors:
    find / -perm -4000 -type f 2>/dev/null
  2. Identify: In the massive output list, you are looking for non-standard binaries. Look closely at /usr/bin/find. It should not normally have the SUID bit set.
  3. Research: Go to GTFOBins online, search for `find`, and look at the "SUID" section.
  4. Exploit: The `find` binary has an `-exec` flag which allows executing arbitrary shell commands on files it finds. Because the binary has SUID set, the `-exec` flag spawns a process as the owner of `find` (root).
    find . -exec /bin/sh -p \; -quit
  5. Verify: Run id. You are now uid=0(root) euid=0(root)!

Phase 3: The SUDO Attack Vector

If the SUID binary route fails, the next best check is what commands you are allowed to run as another user using `sudo`.

  1. Enumerate: Run sudo -l.
  2. Identify: Notice the output says: (ALL : ALL) NOPASSWD: /usr/bin/vim. This means you can run the vim text editor as root without needing the root password.
  3. Exploit: Vim has a built-in feature to spawn shells. Run vim with sudo:
    sudo vim
  4. Drop to Shell: Once inside vim, hit ESC to ensure you are in command mode, then type:
    :!/bin/bash and hit Enter.
  5. Verify: Run whoami. You are root again!

Remediation & Hardening Strategies

Privilege escalation paths are considered misconfigurations. Remediation requires strict access controls.

  • SUID Hardening: Use `chmod -s` to remove the SUID bit from binaries that don't absolutely require it. Only `passwd`, `sudo`, `su`, and a few others should have it.
  • Sudoers Auditing: Never grant `NOPASSWD` sudo rights to binaries that can spawn subshells or execute OS commands (e.g., `tar`, `awk`, `vim`, `find`, `less`, `more`, `python`).
  • Principle of Least Privilege: Implement strict MAC (Mandatory Access Controls) using AppArmor or SELinux to sandbox the web applications so even if they are compromised, they cannot read `/etc/crontab` or `sudoers`.
