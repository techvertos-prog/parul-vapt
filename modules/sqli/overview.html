Module: SQL Injection (SQLi) Deep Dive

1. Core Architectural Concept 🧠

SQL Injection (SQLi) is one of the oldest and most devastating vulnerabilities on the web. It occurs when an application improperly sanitizes user input before concatenating it directly into a backend database query. This allows an attacker to manipulate the structure of the query itself, enabling them to read, modify, or delete data they shouldn't have access to.

The core problem isn't the database—it's the String Concatenation in the application layer (PHP, Java, Node.js). If the database thinks your input string is an instruction rather than data, the application is fundamentally compromised.

💡
Pro Tip: Almost all SQL Injection stems from a developer "building queries manually using plus signs or dot operators" instead of using secure Parametrized Queries.

2. The Three Families of SQLi 🎯

The way you exploit SQLi heavily depends on how much information the application returns to you. We categorize them into three families.

Family 1: Error-Based / In-Band (The Easy Way)

The application takes your payload, the DB crashes, and the application prints the raw database error on the screen. Or, you use UNION statements to make the application display the database contents directly on the webpage.

👨💻 Attacker inputs `?id=1'`
↓
🗄️ Database Query: `SELECT * FROM users WHERE id=1'` (Syntax Error)
↓
💉 Webpage displays: "You have an error in your SQL syntax near '''"

Family 2: Boolean-Based Blind (The Guessing Game)

The application does NOT display database errors and does NOT display query results. However, if a query evaluates to TRUE, the page looks normal. If a query evaluates to FALSE, the page looks slightly different (e.g., a missing 'Welcome' message). You steal data by asking a million True/False questions.

👨💻 Attacker inputs `?id=1 AND (SELECT SUBSTRING(version(),1,1))='5'`
↓
🗄️ Database checks if the first number of its version is 5.
↓
💉 Webpage returns 'Welcome' (True). The attacker now knows the DB version starts with 5.

Family 3: Time-Based Blind (The Waiting Game)

The hardest type. The application behavior does not change at all whether the query is true or false. You steal data by telling the database to pause for 5 seconds if a character matches your guess. If the server takes 5 seconds to respond, your guess was correct.

👨💻 Attacker inputs `?id=1; IF (SYSTEM_USER='sa') WAITFOR DELAY '0:0:5'--`
↓
💉 Page takes exactly 5 seconds to load. Attacker knows the user is 'sa' (System Admin).

Phase 1: The Authentication Bypass (Manual Execution)

Many legacy login forms take a username and password and do a simple string match in the database.

// Vulnerable Code
$query = "SELECT * FROM users WHERE username='" . $_POST['user'] . "' AND password='" . $_POST['pass'] . "'";
  1. Go to http://localhost:8080/login.php in the lab environment.
  2. In the Username field, enter the classic payload: admin' OR '1'='1' #
  3. Enter anything in the password field, and click Login.
  4. Why it works: The query becomes: SELECT * FROM users WHERE username='admin' OR '1'='1' #' AND password='xyz'. The # comments out the password requirement entirely, and 1=1 is always true, so the database logs you in as the first user it finds (usually the Admin).

Phase 2: Database Extraction (Automated via SQLMap)

Manual extraction via UNION or Blind techniques is tedious. In a real VAPT scenario, we rely on automation tools like SQLMap to dump massive tables once we identify an injection point.

  1. Find a vulnerable URL parameter that fetches data. For example: http://localhost:8080/item.php?id=1
  2. Open your terminal and run SQLMap to enumerate the available databases:
    sqlmap -u "http://localhost:8080/item.php?id=1" --dbs --batch
  3. SQLMap will discover the backend is MySQL, test all 3 families of injection, and return a list of databases (e.g., `app_db`).
  4. Dump the entire users table, bypassing the application logic completely:
    sqlmap -u "http://localhost:8080/item.php?id=1" -D app_db -T users --dump --batch

Remediation & Hardening Strategies

⚠️
WAFs are not the answer! Blocking quotes (`'`) or keywords (`SELECT`) using a Web Application Firewall will eventually be bypassed. The only true fix happens in the source code.

Always use Prepared Statements (Parameterized Queries) or ORMs (Object-Relational Mapping). This completely forces the database engines to treat the variables as raw string data, not executable SQL syntax.

/* SECURE CODE (PHP PDO Example) */
$stmt = $pdo->prepare("SELECT * FROM users WHERE username = :username AND password = :password");
$stmt->execute(['username' => $_POST['user'], 'password' => $_POST['pass']]);
$user = $stmt->fetch();
